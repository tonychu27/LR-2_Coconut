LR-2 Coconut on GSM8K with Qwen3-0.6B-Base

This report describes my implementation of Coconut-style continuous reasoning for GSM8K, the debugging issues I hit while making the latent feedback path train correctly, and the evidence from a one-example overfit run and the GSM8K evaluation artifacts currently available in runs/.

What I Built

I implemented a compact Coconut reproduction around Qwen/Qwen3-0.6B-Base. The repo contains a standard chain-of-thought baseline, a Coconut latent-reasoning model, LoRA-only training paths, and a unified evaluator. The central idea is to replace part of the written reasoning trace with continuous vectors instead of sampled text tokens.

CoT Baseline

The CoT model sees the problem prompt and learns to generate textual reasoning followed by #### answer. This is the ordinary decoder-only language modeling setup.

Coconut

The Coconut model sees the prompt ending with <bot>, performs several latent reasoning steps by feeding hidden states back as embeddings, then receives <eot> and decodes the answer in normal text.

How Coconut Works

For a normal language model, every next position receives an embedding that comes from a token id. Coconut changes that only for a short latent segment. After the prompt and the <bot> marker, I run the base model once, take the last layer hidden state at the final position, and concatenate that hidden state directly onto the input embedding sequence as the next position. I repeat this for latent_steps. No token is sampled inside the latent segment, and no tokenizer lookup happens there. 

prefix text + <bot>
  -> model forward
  -> last hidden state h_1 becomes next input embedding
  -> model forward
  -> last hidden state h_2 becomes next input embedding
  -> ...
  -> append <eot>
  -> generate visible answer text

This matters because the latent states are continuous model-internal vectors. They can carry information that does not have to be compressed into a single discrete token at every step. During training, those fed-back hidden states remain connected to the computation graph, so gradients can flow through the continuous reasoning path.

Curriculum

The GSM8K answers include rationales followed by the final #### answer. I split each rationale into reasoning steps. The curriculum progressively hides the leading reasoning steps from the text target. At stage s, the first s reasoning steps are removed from the suffix, and the latent segment must carry the information that those text steps used to provide. 

stage 0 target:
  <eot>
  step 1
  step 2
  step 3
  #### answer

stage 1 target:
  <eot>
  step 2
  step 3
  #### answer

stage 2 target:
  <eot>
  step 3
  #### answer

This makes the task harder gradually. The model first learns the ordinary answer format and later learns to use the continuous latent states in place of some visible reasoning text.

Problems Hit And How I Solved Them

Problem Why It Mattered Solution
Sequential latent forward pass The latent embedding at step t+1 is not known until the model has produced the hidden state at step t. This means the latent region cannot be prepared as a static tensor of token embeddings. I implemented the latent segment as a loop. Each iteration runs the base model with inputs_embeds, extracts hidden_states[-1][:, -1:, :], and appends it to the embedding sequence.
Gradient flow through fed-back hidden states If the hidden state is detached, converted to token ids, sampled, or passed through an argmax operation, the latent pathway stops being differentiable. The hidden state tensor is concatenated directly into the next inputs_embeds. I avoid detach(), token sampling, and token-id conversion in the latent segment.
Training speed and memory Recomputing the whole prefix for every latent step is simple but slow. However, using a cache incorrectly can accidentally detach the hidden-state feedback or prevent gradients from flowing through the latent loop. I kept the reference path explicit and added a cached variant for training/evaluation that still feeds hidden states as tensors. The main correctness condition is that the latent states remain part of the computation graph.
Tokenizer and marker handling If <bot>, <eot>, and <latent> are not stable tokenizer entries, the model may split them into unrelated sub-tokens or fail to align the prompt/suffix boundary. I add the markers as special tokens, set a pad token when needed, and resize the model embeddings. The Coconut config stores the token ids for the markers.
Loss masking It is easy to accidentally train the model to predict prompt tokens or latent positions. That would make the loss look meaningful while optimizing the wrong target. Labels for prefix tokens and latent positions are set to -100. Only selected suffix tokens are supervised. For answer-only loss, visible reasoning tokens are also masked and only the final answer suffix is trained.
LoRA training stability The Coconut LoRA model was sensitive to optimization settings. Without LoRA dropout and a dynamic learning-rate schedule, training could overfit the visible answer pattern or become unstable across curriculum stages, making it hard to get a useful checkpoint. I used LoRA dropout plus warmup and decay for the learning rate. This made training less brittle than a fixed-rate run and helped stage 0 stay close to the CoT LoRA baseline, even though later latent stages still degraded.
Curriculum scheduling Jumping straight to hidden reasoning makes the small model unstable, while training only on visible reasoning does not test the latent pathway. The stage results showed this clearly: stage 0 reached 49.05%, stage 1 dropped to 31.08%, and stage 2/final dropped to 14.48%. I evaluated the saved checkpoints separately instead of only reporting the final checkpoint. This showed that stage 0 is best for accuracy, while stage 1 is the first more meaningful latent-reasoning checkpoint. I report the stage behavior explicitly rather than hiding the collapse behind the final checkpoint.
Reasoning-step parsing GSM8K rationales are often stored as one long line. My initial splitter treated each non-empty line as one reasoning step, so many examples had only one large step. When stage 1 or stage 2 removed leading steps, it often removed almost the whole visible rationale. I diagnosed this by comparing checkpoint outputs. Stage 2/final had a median generation length of only 4 words, and most outputs began with The answer is. This explains why later stages behaved like answer-only guessing instead of successful latent reasoning.
Answer extraction Qwen sometimes emits <answer>18</answer> even when asked for #### 18. The original extractor missed these and undercounted correct answers. I extended extraction to prefer the last explicit <answer>...</answer> tag and then fall back to the last #### answer.

Overfit-One-Example Evidence

I ran the required one-example overfit with Qwen3-0.6B-Base, 2 latent steps, and LoRA. The current artifact in runs/one_example uses 300 optimizer steps. Its logged loss went from 1.9609375 at step 1 to 0.0000050366 at step 300. 

python scripts/overfit_one.py \
  --model Qwen/Qwen3-0.6B-Base \
  --output-dir runs/one_example \
  --latent-steps 2 \
  --max-steps 300 \
  --lr 2e-4 \
  --lora

Conclusion

The one-example overfit confirms that the continuous-thought path can learn through fed-back hidden states and reproduce a GSM8K rationale/answer. The hardest parts were not the final generation loop, but the mechanics around sequential latent computation, keeping gradients alive, aligning special markers, masking the correct labels, and scheduling the curriculum so the model gradually learns to replace text reasoning with latent reasoning.

The corrected original-model evaluation now behaves as expected: direct zero-shot prompting is weak, while 4-shot CoT reaches 60.53% accuracy and closely matches the reported Qwen3-0.6B-Base GSM8K result. The CoT LoRA and Coconut LoRA comparison is still pending because the full Hugging Face evaluation summaries are not available in the current runs/ artifacts.
